# ADR-021: Universal Business Object Model (UBOM)

**Status:** Accepted  
**Date:** 2026-06-30  
**Deciders:** CTO, Platform Team  
**Tags:** architecture, data-model, platform

## Context

SalesOS has multiple entity types (Company, Contact, License, Branch, Deal) that share common concerns:
- Every entity needs a UUID primary key, tenant isolation, timestamps
- Every entity needs Timeline (event history), Audit, Search (full-text + semantic), Knowledge Graph representation, Feature Store integration, and Permissions
- These cross-cutting concerns were being implemented ad-hoc per entity, leading to duplication and inconsistency
- The Capability Framework needs a canonical entity registry so Company360, Deal360 etc. can be generated dynamically

## Decision

Adopt a **Universal Business Object Model (UBOM)** where every business entity inherits from a `BusinessObject` base class.

### Architecture

```
BusinessObject (mixin base)
├── id: UUID (PK)
├── tenant_id: UUID (FK → tenants)
├── status: String
├── owner_id: UUID
├── tags: JSONB
├── metadata: JSONB
├── created_at, updated_at: DateTime
│
├── CompanyObject
├── ContactObject
├── LicenseObject
├── BranchObject
├── DealObject
└── (future entities extend the same base)
```

### What the base provides

| Concern | Mechanism |
|---------|-----------|
| Identity | UUID primary key, auto-generated |
| Tenant isolation | `tenant_id` FK on every entity |
| Status lifecycle | Active/Inactive/Archived etc. |
| Timestamps | `created_at` / `updated_at` with DB defaults |
| Timeline | Via TimelineRuntime wildcard event subscriber |
| Audit | Via EventRuntime audit trail |
| Search | Via SearchRuntime (full-text + semantic) |
| Knowledge Graph | Via KnowledgeGraphEngine (Neo4j auto-population) |
| Feature Store | Via FeatureStore (precomputed scores by entity ID) |
| AI/Decisions | Via DecisionEngine (context + policies) |
| Permissions | Via PolicyRuntime (policy evaluation on entity) |
| Events | Via EventRuntime (CRUD events auto-published) |

### Entity Registration

Every entity class calls `cls.register()` which adds it to `REGISTERED_ENTITY_TYPES` — a global registry consumed by the Capability Framework. This enables dynamic discovery of all entity types.

## Consequences

### Positive
1. **Zero-boilerplate** for new entity types — just inherit and define domain-specific columns
2. **Guaranteed consistency** in cross-cutting concerns across all entities
3. **Dynamic discovery** — Capability Registry, Search, Navigation all read from the same entity registry
4. **Future-proof** — adding a new entity is a single class definition

### Negative
1. Tight coupling — every entity *must* use the base, no opt-out
2. Migration complexity — existing tables must be retrofitted (but they largely already match the base shape)

### Mitigations
- The base provides *capabilities* (Timeline, Search, etc.) but doesn't force their usage — they're opt-in via runtime wiring
- `BusinessObject` is a **mixin**, not a concrete table — SQLAlchemy joins work normally

## Compliance Check

- [x] Every entity has UUID PK, tenant_id, status, timestamps
- [x] Entity registry feeds Capability Framework
- [x] No hardcoded entity lists in navigation/search/permissions
- [x] Timeline works for any BusinessObject via `entity_type` + `entity_id` mapping
